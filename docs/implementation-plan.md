# 토닥 MVP 구현 계획

## 목적

이 문서는 `토닥`의 첫 번째 MVP를 `1주일 내 구현`하는 기준으로 정리한 실행 계획이다.

핵심 목표는 다음 하나다.

> 사용자가 자기 전 `원버튼`으로 5~7분간 대화하고, 다음 날 에이전트가 전날 감정 맥락을 짧게 기억한 상태로 다시 말을 걸어주는 경험을 만든다.

이 계획은 `Next.js + LiveKit + LangGraph` 기반으로 작성했다.

---

## 1. MVP 범위

### 반드시 포함

- PWA 웹앱
- 원버튼 세션 시작
- LiveKit 기반 실시간 음성 연결
- 공감형 음성 대화
- 5~7분 자동 종료
- 대화 종료 후 요약 메모리 저장
- 다음 세션에서 전날 요약 주입
- 기본 안전 응답
- 운영용 최소 로그

### 제외

- 복수 음성 선택
- 회원가입/소셜 로그인
- 대화 원문 열람
- 감정 리포트
- 다국어 지원
- 복잡한 개인 설정
- 다중 서브 에이전트 퍼레이드형 UX

---

## 2. 제품 성공 기준

### 출시 기준

- 첫 세션에서 사용자가 실제로 음성 대화를 끝까지 완료할 수 있다.
- 에이전트 응답이 대체로 `공감형`, `비판단형`, `짧은 문장` 톤을 유지한다.
- 세션 종료 후 원문 없이 요약만 저장된다.
- 다음날 "어제 힘들었다고 했잖아" 수준의 짧은 회상이 가능하다.
- 세션/요약/에러를 운영자가 조회할 수 있다.

### 1주 MVP KPI

- 첫 세션 완료율 `70%+`
- 평균 세션 길이 `3분+`
- 음성 왕복 체감 지연 `2초 이하`
- memory recall 성공률 `80%+`
- 치명적 크래시 없는 상태로 내부 테스트 `20회+`

---

## 3. 기술 전략

### 아키텍처 원칙

- `Next.js`는 UI, 세션 시작, 운영 콘솔 담당
- `Python Agent`는 실시간 대화와 LangGraph 실행 담당
- `Supabase/Postgres`는 최소 상태 저장
- 원문 음성/전체 transcript는 저장하지 않는다

### LangGraph 원칙

- `오케스트레이터`를 거대한 범용 에이전트로 만들지 않는다
- 상태 그래프는 `대화`, `수면 유도`, `메모리 요약`, `안전 응답` 4가지 책임으로 분리한다
- 모든 응답은 "문제 해결"보다 "감정 수용과 수면 유도"를 우선한다

## 4. 1주 구현 일정

## Day 1 - 프로젝트 골격

목표: 전체 시스템이 붙을 수 있는 최소 뼈대를 만든다.

- Next.js 앱 초기화
- PWA 기본 설정
- Supabase 프로젝트/환경변수 설정
- Python agent runtime 초기화
- LiveKit room/token 발급 API 추가
- 공통 환경변수 문서화

완료 기준:

- 브라우저에서 세션 시작 버튼을 누르면 LiveKit 연결이 열린다.

## Day 2 - 음성 입출력 연결

목표: 음성으로 한 턴 대화가 가능해야 한다.

- 마이크 입력 연결
- STT 스트리밍 연결
- LLM 응답 생성
- TTS 스트리밍 재생
- 기본 turn-taking 구현

완료 기준:

- 사용자가 말하면 에이전트가 짧게 대답한다.

## Day 3 - LangGraph 대화 흐름

목표: 공감형 대화 흐름을 그래프로 고정한다.

- `ConversationNode` 구현
- `SafetyNode` 구현
- `SleepTransitionNode` 구현
- 세션 타이머 추가
- 5~7분 종료 흐름 추가

완료 기준:

- 대화 후반부 응답이 짧아지고, 세션이 자동 종료된다.

## Day 4 - 기억 기능

목표: 원문 저장 없이 기억 경험을 만든다.

- 세션 종료 시 summary 생성
- 요약만 DB 저장
- 다음 세션 시작 시 최근 memory 주입
- memory prompt 고정

완료 기준:

- 다음 세션에서 어제 대화의 감정/주제를 1회 이상 회상한다.

## Day 5 - 안정화

목표: 실제 사용 가능한 수준으로 끊김과 불쾌한 반응을 줄인다.

- 침묵 감지
- 응답 중첩 방지
- timeout/retry
- fallback 응답
- 기본 safety 문구
- 로그 정리

완료 기준:

- 내부 테스트에서 세션 중단 없이 5회 연속 동작한다.

## Day 6 - 운영/배포

목표: 테스트 가능한 배포본을 만든다.

- Vercel 배포
- agent 서버 배포
- 운영용 세션 목록 페이지
- 요약/에러 조회 화면
- Sentry 추가

완료 기준:

- 모바일에서 홈 화면 추가 후 실제 대화 가능
- 운영자가 최근 세션과 요약을 볼 수 있다

## Day 7 - 내부 베타

목표: 기능 추가가 아니라 품질 개선만 한다.

- 테스트 대화 시나리오 20개 실행
- 첫 30초 UX 튜닝
- 음성 톤/속도 조정
- safety 문구 검토
- 리텐션용 푸시/알림은 보류

완료 기준:

- 5~10명에게 공유 가능한 베타 링크 확보

### P1

### P2

---

## 6. 팀 없는 개발 방식

솔로프리너 기준으로 `병렬 작업`보다 `빠른 검증 루프`가 중요하다.

- 아침: 전날 실패 세션 리뷰
- 낮: 구현
- 밤: 실제 침대 환경 테스트

매일 확인할 것:

- 응답이 사람처럼 다정한가
- 응답이 길어서 각성시키지 않는가
- 회상 문장이 부담스럽지 않은가
- 음성 자체가 다시 듣고 싶을 정도인가

---

## 7. 런칭 전 체크리스트

- 개인정보 처리 문구 작성
- 원문 미저장 정책 반영
- 위기 상황 안내 문구 반영
- session timeout 정책 반영
- 에러 시 "텍스트 화면"이 아니라 "짧은 음성 사과"로 복구

---

## 8. 구현 후 바로 판단할 것

- TTS 원가가 월 구독 모델을 버틸 수 있는가
- 5분이 맞는지 7분이 맞는지
- 무료 티어를 줄지 말지
- 실제 유저가 `위로`를 원하는지 `잠 신호`를 원하는지

현재 단계에서는 기능보다 아래 질문에 빨리 답하는 게 중요하다.

> "사람이 이 목소리를 자기 전에 매일 다시 켤 것인가?"
